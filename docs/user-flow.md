│          欢迎消息                    │  └──────────────────────┘    │
│   「你好！我是你的 AI 购物助手...    │                              │
│     告诉我你想找什么样的耳机」       │  需求标签: (空)               │
│                                     │  候选漏斗: (空)               │

**系统状态：**
- 向量索引已加载（首次启动约 30 秒构建，后续从缓存加载约 2 秒）

- 等待用户输入

---

### Step 1：用户输入需求

**用户操作：** 在底部输入框中输入自然语言需求

**典型输入示例：**
- "帮我找一款 500 以内的降噪耳机，主要通勤用"
- "推荐个游戏耳机，头戴式的"
- "有没有运动时戴的蓝牙耳机，200 左右"
- "我想买个耳机"（信息不足，系统会追问）

**系统行为：**
1. 前端通过 POST `/api/chat/stream` 发送消息 + 历史记录
2. 后端将消息传递给 LangGraph Agent
3. Agent 分析用户意图，提取已知约束条件
4. Agent 判断信息是否充足（"预算+类型"或"预算+场景"即可），决定追问还是直接搜索

---

### Step 2：意图澄清（多轮对话 + 快捷回复）

**用户看到的：**

```
👤 帮我找一款500以内的降噪耳机，主要通勤用

🤖 [意图澄清] 好的！500元以内的降噪耳机，通勤场景。
   通勤场景入耳式更主流，我默认按入耳式找，你觉得呢？

   [ 入耳式 ]  [ 头戴式 ]  [ 都可以 ]    ← 快捷回复按钮
```

**决策面板状态：**
```
进度: ● 了解需求 → ○ 搜索商品 → ○ 比较推荐 → ○ 确认下单

需求标签: [预算≤500] [降噪] [通勤]
```

**系统行为：**
- Agent 通过 SSE 逐字流式输出回复
- Agent 输出 `quick_replies` JSON，前端渲染为快捷回复按钮
- Agent 主动给出默认建议（如"通勤场景入耳式更主流"），减少交互轮次
- 不会重复追问用户已回答过的问题

**分支逻辑：**
- 如果用户信息很少（如"我想买个耳机"），Agent 追问预算和场景
- 如果用户给了充分信息，Agent 可能直接跳到搜索阶段
- 快捷回复和手动输入都被接受

---

### Step 3：用户补充信息

**用户操作：** 点击快捷回复按钮或手动输入

```
👤 入耳式    ← 点击快捷回复按钮
```

**系统行为：**
- Agent 更新约束条件：类型 = 入耳式
- Agent 判断信息充足，进入任务计划阶段

   │ #  │ 步骤     │ 说明                      │
   ├────┼──────────┼──────────────────────────┤

**决策面板状态：**
```
进度: ✅ 了解需求 → ● 搜索商品 → ○ 比较推荐 → ○ 确认下单

需求标签: [预算≤500] [降噪] [通勤] [入耳式]
```

### Step 5：商品检索 + 比价 + 库存（工具调用阶段）

**用户看到的：**

工具调用过程实时展示，可展开查看详情：

```
🤖 [搜索中]

   🔧 调用了 3 个工具                        ▼ 展开
   ┌───────────────────────────────────────────────┐
   │ ✅ product_search("500元入耳降噪耳机 通勤")     │
   │    → 检索到 8 条匹配商品                       │
   │ ✅ compare_prices("小米 Redmi Buds 5 Pro")     │
   │    → 京东¥319 / 天猫¥299 / 拼多多¥279         │
   │ ✅ check_inventory("小米 Redmi Buds 5 Pro",    │
   │    "拼多多")  → 库存: 56 件                    │
   └───────────────────────────────────────────────┘
```

**决策面板更新：**
```
候选漏斗:
  500 条商品 → 8 条检索 → 价格筛选 → 类型筛选 → 3 条推荐

比价表格:
  商品A: 京东¥319 | 天猫¥299 | 拼多多¥279
  商品B: 京东¥459 | 天猫¥449 | 拼多多¥439
  商品C: 京东¥499 | 天猫¥509 | 拼多多¥489
```

**系统行为：**
1. Agent 调用 `product_search` — 语义向量检索 Top-8 匹配商品
2. Agent 调用 `compare_prices` — 获取候选商品在京东/天猫/拼多多的价格
3. Agent 调用 `check_inventory` — 查询推荐商品的库存状态
4. 每个工具调用通过 SSE `tool_start` / `tool_end` 事件实时通知前端
5. Agent 根据用户约束从 8 条中筛选出 3 条最佳推荐

   │ Buds 5 Pro        │ │ FreeBuds 5i       │ │ NeoBuds Pro 2     │
   │                   │ │                   │ │                   │
   │ ¥279 (拼多多最低)  │ │ ¥439 (拼多多最低)  │ │ ¥489 (拼多多最低)  │
   │ ⭐ 4.4            │ │ ⭐ 4.5            │ │ ⭐ 4.6            │

**决策面板状态：**
```
进度: ✅ 了解需求 → ✅ 搜索商品 → ● 比较推荐 → ○ 确认下单
```

**分支逻辑 — 用户可能的回应：**

| 用户回应 | 系统处理 |
|---------|---------|
| 点击 "选择这款" | → 进入 Step 7（订单确认弹窗） |
| "有没有更便宜的？200以内" | → Agent 更新预算，重新搜索（回到 Step 5） |
| "头戴式的有吗？" | → Agent 更新类型，重新搜索（回到 Step 5） |
| "第二款和第三款再详细对比一下" | → Agent 展开对比分析，留在当前步骤 |
| "这几个都不喜欢，有没有 JBL 的" | → Agent 更新品牌偏好，重新搜索（回到 Step 5） |

---

### Step 7：订单确认（弹窗）

**用户操作：** 点击商品卡片上的"选择这款"

**用户看到的：**

弹出模态确认窗口：

```
┌───────────────────────────────────┐
│         📦 确认订单信息             │
│                                   │
│   商品：小米 Redmi Buds 5 Pro      │
│   价格：¥279                      │
│   平台：拼多多                     │
│   库存：有货                       │
│   预计送达：明天 18:00 前           │
│                                   │
│      [ 取消 ]    [ 确认下单 ]      │
└───────────────────────────────────┘
```

**决策面板状态：**
```
进度: ✅ 了解需求 → ✅ 搜索商品 → ✅ 比较推荐 → ● 确认下单
```

**系统行为：**
- Agent 输出 `order_confirm` 结构化 JSON（此时不调用下单工具）
- 前端渲染为弹窗，展示订单详情
- 价格数据从商品缓存中校验，防止 LLM 幻觉导致价格错误

**分支逻辑：**

| 用户回应 | 系统处理 |
|---------|---------|
| 点击 "确认下单" | → 进入 Step 8（执行下单） |
| 点击 "取消" | → 关闭弹窗，回到推荐方案 |
| "换京东的" | → Agent 更新平台和价格，重新展示确认信息 |

## 特殊场景

### 场景 A：用户中途改需求

```
👤 帮我找降噪耳机，500以内
🤖 [意图澄清] 通勤场景入耳式更主流...
   [ 入耳式 ]  [ 头戴式 ]
👤 入耳式
🤖 [推荐方案] 推荐3款... (卡片展示)
👤 预算放宽到1000，有什么更好的？
🤖 [搜索中] 好的，为你重新搜索1000元以内的...
🤖 [推荐方案] 新的3款推荐... (卡片展示)
```

Agent 会感知需求变化并说明"预算从500调整到1000"，而不是从头重新开始。

### 场景 B：用户需求模糊

```
👤 我想买个耳机
🤖 [意图澄清] 帮你挑耳机！先了解一下：
   预算大概多少？主要什么场景用？
   [ 通勤 ]  [ 运动 ]  [ 游戏 ]  [ 办公 ]
👤 通勤用，300左右
🤖 [意图澄清] 通勤降噪体验会好很多，需要降噪功能吗？
   [ 要降噪 ]  [ 不需要 ]
👤 要降噪
🤖 [分析中] 📋 任务计划: ...
```

Agent 通过多轮追问 + 快捷回复逐步明确需求。

### 场景 C：用户直接选择不下单

```
👤 帮我找游戏耳机
🤖 [意图澄清] ...
👤 头戴式，500以内
🤖 [推荐方案] 推荐3款... (卡片展示)
👤 谢谢，我先了解一下
🤖 好的，有需要随时找我！
```

用户无需下单即可结束对话。

## 界面功能说明

### 顶部 Header

| 功能 | 说明 |
|------|------|
| LLM 模式选择 | 切换 Ollama 本地 / Qwen 云端 API |
| 清空对话 | 重置所有对话记录和决策面板状态 |

### 左侧聊天面板

| 功能 | 说明 |
|------|------|
| 流式输出 | AI 回复逐字出现，实时感 |
| 阶段标签 | 每条消息标注当前阶段（如 [推荐方案]） |
| Markdown 渲染 | 支持表格、列表、加粗等格式 |
| 工具调用日志 | 可展开查看 AI 调用了什么工具、返回了什么结果 |
| 商品卡片 | 3 张推荐卡片横向排列，点击选择 |
| 快捷回复 | 按钮式快捷回复，减少用户输入 |
| 订单确认弹窗 | 模态窗口展示订单详情，确认/取消 |

### 右侧决策面板（可拖拽调整宽度）

| 功能 | 说明 |
|------|------|
| 阶段进度条 | 了解需求 → 搜索商品 → 比较推荐 → 确认下单，四步可视化 |
| 用户需求标签 | 提取的需求以标签形式展示（预算、类型、场景等） |
| 候选漏斗图 | 500条 → 检索结果 → 价格筛选 → 类型筛选 → 3条最终推荐 |
| 跨平台比价表格 | 候选商品在京东/天猫/拼多多的价格对比 |

决策面板是演示亮点 — 让观众直观看到 AI 的决策过程和数据依据。
